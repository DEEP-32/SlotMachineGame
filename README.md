# Slot Machine Development Journal

This document outlines the technical approach, architectural decisions, and problem-solving journey during the development of this Unity-based slot machine system.

## 1. Architectural Overview

The project follows a **Separation of Concerns** pattern, ensuring that game logic (math), UI management, and visual animation remain decoupled.

* **SlotMathEngine:** The "Brain." Pure C# logic that handles weighted probability calculations and win evaluation, completely independent of Unity's visual systems.

* **ReelManager:** The "Orchestrator." Manages the sequence and timing of reel stops to create professional suspense.

* **ReelController:** The "Executor." Handles the specific movement logic, symbol wrapping, and state management for individual columns.

* **Data Layer:** Utilizes `ScriptableObjects` for `SymbolDatabase` and `SlotSymbol` data, allowing designers to tweak game balance without touching code.

## 2. Key Problem-Solving Approaches

### Problem A: Visual "Ghosting" and Alignment (The Stop Logic)

**Initial Challenge:** The reels would often stop with symbols misaligned because the `Update()` loop was checking for a floating-point threshold (`actualY <= 0`) while the reel continued to move, leading to inconsistent offsets.

**The Solution:**
Instead of checking frame-by-frame float values, we transitioned to a **deterministic sequence approach**:

1. We decoupled the movement from `Update()` and moved it to a `Coroutine` (step-based movement).

2. We implemented a `stoppingSequence` queue. When the queue is empty, the reel is mathematically guaranteed to have completed its target rotations.

3. We added a hard reset to `Vector2.zero` at the moment of completion, ensuring the symbol is perfectly centered in the `GridLayoutGroup` every time, regardless of frame rate.

### Problem B: Synchronization between Math and Visuals

**Initial Challenge:** Ensuring the result generated by the `SlotMathEngine` matched what the player saw on screen.

**The Solution:**
We established a strict **Command-Pattern** flow:

* The `GameManager` asks the `SlotMathEngine` for a result (`SpinResult`).

* This result is passed directly to `ReelManager.StopReels`.

* The `ReelController` is explicitly *commanded* to land on specific symbols using `StopOnSymbol(targetType)`.

* This removes randomness from the ReelController's stop phase, making the visual result a direct puppet of the math engine's output.

### Problem C: Web Build Compatibility

**Initial Challenge:** The original save system used `System.IO.File`, which is incompatible with browser sandboxes in WebGL builds.

**The Solution:**
We refactored the `SaveHandler` to utilize Unity’s `PlayerPrefs` (which acts as a wrapper for IndexedDB in browsers). This ensures progress is saved across browser sessions without violating security constraints.

## 3. Technical Highlights

* **Dynamic UI Scaling:** Used `GridLayoutGroup` and `LayoutRebuilder` to ensure the reel automatically calculates symbol spacing based on prefab dimensions, making the system resolution-independent.

* **Tweening/Animation:** Created a modular `ReelAnimation` component that handles the "snap-and-bounce" effect. By passing callbacks (`Action`), we ensure the system waits for the animation to finish before notifying the manager that the spin is fully complete.

* **Rich Text Integration:** Implemented TextMeshPro rich text tags (`<color>`) to allow for dynamic, multi-colored UI elements (e.g., "Bet 10 G") within a single text object, optimizing draw calls.

## 4. Future Scalability

The system is built to be easily extended:

* **Adding Symbols:** Simply update the `SymbolDatabase` ScriptableObject; no code changes required.

* **New Payout Rules:** Update the `EvaluateWin` method in `SlotMathEngine` without needing to adjust any visual UI logic.
